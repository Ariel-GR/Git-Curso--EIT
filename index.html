Configuración inicial

Ajustes globales

Se aplicarán a todos los proyectos

  • git config --global user.name <usuario> para definir un usuario
  • git config --global user.email <email> para definir un email. Ni el email ni el usuario tienen relación con los datos de gihub.com
  • git config --global core.editor <editor> para definir qué editor de texto usar con git
  • git config alias.<alias> <command> se utiliza para definir un atajo a un comando de git.

Areas de GIT

Working Directory

El area de trabajo corresponde a la carpeta donde inicializamos el repositorio de git y realizamos la creacion, edicion y eliminacion de archivos en nuestro proyecto. En esta instancia haremos muy poco uso de la linea de comandos, mas alla de algunas instrucciones basicas.

  • dir || ls muestra el contenido del directorio actual
  • mkdir <name> crea una carpeta con el nombre especificado
  • cd <path> cambia el directorio actual por el seleccionado
  • git init inicializa un respositorio de git.
  • rm -rf .git elimina el repositorio de manera forzada

Staging Area (INDEX)

El area de control de cambios corresponde a un archivo que registra a que elementos le estamos haciendo seguimiento dentro de nuestro proyecto. Cada vez que agregamos un archivo al INDEX se realiza una captura del codigo (SNAPSHOT) que registra los cambios realizados; una vez realizada la misma, podemos descartarla o confirmarla.

  • git status muestra el estado de los archivos con respecto al INDEX, si se esta haciendo seguimiento de los mismos o no, y los posibles resultados son los siguientes:
    • untracked no se esta haciendo seguimiento y por lo tanto es irrecuperable en caso de perdida
    • staged el archivo fue agregado al INDEX, pero aun no se realizo la confirmacion de cambios
    • commited los cambios del archivo fueron confirmados en un commit y pueden ser reestablecidos
    • modified el archivo difiere con respecto a la ultima confirmacion realizada
  • git diff compara los cambios del Working Directory con respecto a los almacenados en el INDEX
  • git restore --staged <file> deshace la ultima captura realizada en el INDEX
  • git rm --cached <file> quita el archivo del area de control de cambios

Repository

El repositorio corresponde a la carpeta .git, donde se almacenan todos los cambios en formato BLOB(Binary Large Object File) utilizando referencias de los nombres implementados en el proyecto. Tambien se utiliza para navegar entre los commits como puntos de control.

  • git commit realiza la confirmacion de cambios de las capturas realizadas
  • git blame <file> muestra el contenido del archivo identificando al autor de cada cambio
  • git log muestra el historial de confirmacion de forma descendente
  • git checkout <ref> permite movernos dentro del historial de cambios
  • git fetch <remote> <branch> captura los cambios realizados en un repositorio remoto
  • git push <remote> <branch> envia los cambios realizados a un repositorio remoto

Apuntadores de GIT

HEAD

Apuntador Movil que se utiliza para indicar al usuario donde se encuentra actualmente, permite la navegacion dentro del historial de cambios e incluso lo podemos utilizar como referente mediante expresiones como HEAD^ o HEAD~1 para indicar a cuantos ancestros o commits anteriores hacemos referencia.

  • git diff HEAD compara el contenido del working directory con el ultimo commit emitido.
  • git checkout HEAD~1 mueve el apuntador a la confirmacion anterior a la que nos encontramos actualmente.

BRANCH

Apuntador dinamico que siempre hace refencia al ultimo commit dentro de la historia en la que encontremos, se utilizan para implementar caracteriscas de prueba, realizar revisiones o correcciones, sin modificar la linea principal del proyecto. Tambien facilitan la navegacion dentro del historial de cambios.
